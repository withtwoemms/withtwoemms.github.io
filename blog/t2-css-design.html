The Science of CSS

A bit on the Visual Formatting Model & Positioning Schemes

March 29th, 2015

The use of cascading style sheets--CSS--makes implementing styles for many pages, much easier than it otherwise would be. Rather than changing the stylings on each individual page, they can be changed in one .css file and rendered accordingly on each page. The initialism, CSS, includes the term cascade in reference to the way child elements inherit properties from their parents. While the concept of the cascade is fairly straightforward, how it works in practice can be the source of much frustration. Sometimes properties that should cascade: do not, and those that should not: do. This leads a unexpected stylings and positionings. To take control of your DOM's style, the Visual Formatting Model must be discussed.

The Visual Formatting Model is a parsing algorithm written about at length here and here. The VFM is instituted by the browser which renders a box for each element in the DOM. The box wraps the content--i.e. the text between tags--and through the use of the style tag or attrbute, the way element boxes are rendered can be controlled. The use of stylesheet is an abstracted and surrogate way of doing this. In the majority of CSS applications, there are three main levers of control available: the type of box, the dimensions of the box, and the positioning scheme of the box.

Before considering what impacts how a box is rendered, an explanation is in order. The anatomy of a box is neatly given below: Content is text and it is wrapped by a border. That border can be separated from the content by distance called padding. The background of an element refers to the space within the border boundary. The las aspect of a box is the margin. The margin clears the space between its boundaries and the border. There you have it: "the anatomy of a box" ... moving on...

Boxes can be displayed as either inline- or block-level elements. Inline display of an element is the most natural in that it resembles text in a word processor. An element is rendered "in line" with the text thus negating any specified height attribute. The width of an inline element is still determinable using the "width" attribute". Unlike the inline element the block element is rendered with line breaks on either side (left and right) and so displaces inline elements. The dimensions of the block element are fully tunable. To make a box that responds to text-control attributes (i.e. "text-align") and still has the full dimensional control, a box may be displayed as an inline-block element.

Now that the box and how it can be displayed have been described it would be useful to cover how to position it. A positioning scheme governs how placement of the box is done--one or another may be useful depending on the desired effect. There are three schemes. The default is the normal flow boxes are rendered sequentially within their container. Next is the float scheme wherein a box within some containing box (more on containers another day) is said to have been 'floated' out of the container. 'Floated' boxes exist alongside their containers. Then there is the absolute positioning scheme. In the absolute scheme, the box is no longer within the flow and so does not interact an boxes within the flow.

I don't care if I've convinced you at this point, but there is a 'rhyme and reason' to CSS styling. Issues, frustration, and tedium usually come from a confusion or mixing of positioning schemes, display styles, or both. More of the pitfalls and contradictions later.
